## 2. 行语义

这份数据的基本分析单元是：

```text
设备停留记录 = ad_id + latitude/longitude + [start_time, end_time] + time_spent
```

线性理解如下：

1. `ad_id` 表示一个匿名设备。
2. `latitude` 和 `longitude` 表示该行停留记录的空间中心点。
3. `start_time` 和 `end_time` 表示停留区间的 UTC Unix 秒时间戳。
4. `time_spent` 表示该停留区间的持续秒数。
5. `cluster_id` 是上游数据中的停留点或地点簇编号。
6. `user_timezone`、`start_time_local` 和 `end_time_local` 是该记录对应的本地时区与本地时间表达。

因此，一行数据应该被理解为“某设备在某个固定停留点停留了一段时间”。在这份文件内部，没有提供从 `start_time` 到 `end_time` 之间的移动轨迹，也没有提供该停留区间内部的位置变化。

## 7. 进入阴影计算的线性逻辑

建议按以下顺序处理该文件：

1. 使用显式 schema 读取 CSV。尤其要把 `start_time_local` 和 `end_time_local` 作为字符串保留。
2. 校验必要字段：`ad_id`、`latitude`、`longitude`、`start_time`、`end_time`、`time_spent`、`user_timezone`。
3. 以 `start_time` 和 `end_time` 作为权威 UTC 时间区间。本地时间字段只用于本地日期统计和结果解释。
4. 把每一行理解为一个固定位置上的停留区间，不在区间内插值移动路径。
5. 根据研究口径裁剪时间区间，例如 UTC 日、本地日、白天时段，或其他研究定义的时间窗口。
6. 根据研究口径裁剪空间范围，例如 lower 48、全美国及属地，或自定义研究区。
7. 将有效停留区间转成阴影计算任务。可以使用 cell、H3 或其他空间键做任务调度和复用，但最终几何判断必须回到原始经纬度点。
8. 在完成时间和空间过滤后，再接入建筑物、树冠和天气数据，避免对夜间、区外或无关记录做无效计算。
9. 输出结果中应保留行级追踪键和时长权重，便于后续统计暴露时间，而不是只统计记录数。

## 8. 对 ShadowMap 计算的含义

这份文件的行数是 `30,057,723`，但它代表的停留时间远大于行数本身。按全文件 `time_spent` 求和，约为 `884.38` billion seconds，也就是约 `14.74` billion stay-minutes。也就是说，如果直接逐行、逐分钟做精确阴影求解，计算量会非常大。

因此，面向该文件的生产计算应遵循以下原则：

1. 保持语义准确：一行是固定停留点上的时间区间。
2. 夜间时段不进入精确阴影几何求解。
3. 通过空间和时间任务键复用可共享的阴影结果。
4. 最终点位分类仍使用原始经纬度，不用 cell 中心点替代真实点。
5. 建筑物和树冠候选数据必须先经过分区检索和预筛，再进入精确求解。

当前主要风险不是 CSV 本身的数据质量。核心字段检查显示这份文件是干净的。真正的工程风险是：如果没有充分的时间裁剪、空间分区和结果复用，长停留区间会被展开成过大的点分钟计算量。
